==============================================================================
Skill-as-Anchor 假设检验：基于 SkillsBench 与 SWE-Skills-Bench 的联合分析
==============================================================================

一、研究问题
------------------------------------------------------------------------------

当 Skill（结构化的程序性知识文档）被注入 LLM Agent 的上下文时，它对 Agent
执行轨迹的影响是「锚定式」还是「弥散式」的？

  • 锚定模式（Anchor）：Skill 在少数关键决策点上提供了决定性信息，使 Agent
    从「完全不会」变为「会做」。效果集中在几个 step，表现为 task 级的 binary
    翻转（fail→pass）。

  • 弥散模式（Diffuse）：Skill 让 Agent 在每一步都略好一些，效果均匀分布在
    整条轨迹上，表现为连续的、小幅的质量提升。

本分析基于两个独立基准的公开数据，共计 133 个 task、7,381 条 Agent 轨迹、
7 个前沿模型、11 个领域。

二、数据来源
------------------------------------------------------------------------------

  数据集 1：SkillsBench（BenchFlow, 2026）
    • 84 个 task，覆盖 11 个 domain
    • 7,381 条轨迹（7 个 agent-model 配置 × 3 个条件）
    • 条件：No Skills / With Skills（人工策划）/ Self-Generated Skills
    • 评估：容器化 deterministic verifier（binary PASS/FAIL）
    • Agent：Claude Code, Codex CLI, Gemini CLI
    • 模型：GPT-5.2, Claude Opus 4.5/4.6, Sonnet 4.5, Haiku 4.5,
            Gemini 3 Pro/Flash

  数据集 2：SWE-Skills-Bench（GeniusHTX, 2026）
    • 49 个 task，纯软件工程领域
    • 每个 task 配有一份 skill document（平均 1,160 tokens）
    • 评估：Docker 容器 + pytest 单元测试
    • 论文报告结果：39/49 零增益，7/49 有效（最高 +30%），3/49 有害

三、总体效果
------------------------------------------------------------------------------

  SkillsBench 总体 pass rate：
    No Skills:      24.3%  (n=2940)
    With Skills:    40.0%  (n=3013)
    Self-Generated: 22.3%  (n=1428)
    Δ(With - No):   +15.7%
    Δ(Self - No):   -2.0%

  SWE-Skills-Bench 论文报告：
    平均增益：+1.2%
    有效 skill：7/49（14.3%），最高 +30%
    有害 skill：3/49（6.1%），最低 -10%

  两个 benchmark 的关键差异：SkillsBench 覆盖 11 个 domain（包含医疗、能源
  等模型薄弱领域），平均增益 +15.5%；SWE-Skills-Bench 仅覆盖软件工程（模型
  强项），平均增益仅 +1.2%。这一差异本身就是 anchor 假设的间接证据——模型
  越不懂的领域，skill 的 anchor 效果越强。

四、Per-task Anchor 分析
------------------------------------------------------------------------------

  SkillsBench 84 个 task 的 Δ(With Skill - No Skill) 分布：

    Strong positive (Δ≥50%):    12 tasks (14.3%)
    Moderate positive (20-50%):  12 tasks (14.3%)
    Weak positive (5-20%):       19 tasks (22.6%)
    Neutral (|Δ|<5%):            31 tasks (36.9%)
    Weak negative (-20% to -5%):  9 tasks (10.7%)
    Negative (Δ≤-20%):            1 tasks (1.2%)

  Strong anchor tasks（Δ≥50%，no-skill 几乎全 fail → with-skill 大幅 pass）：

    sales-pivot-analysis                       0% → 81%  (Δ=+81%, n=37+35)
    mario-coin-counting                        3% → 84%  (Δ=+81%, n=37+35)
    flood-risk-analysis                        3% → 80%  (Δ=+77%, n=35+35)
    sec-financial-report                       0% → 73%  (Δ=+73%, n=37+35)
    offer-letter-generator                     14% → 86%  (Δ=+72%, n=37+35)
    earthquake-plate-calculation               11% → 83%  (Δ=+71%, n=35+35)
    protein-expression-analysis                17% → 86%  (Δ=+69%, n=37+35)
    manufacturing-fjsp-optimization            0% → 69%  (Δ=+69%, n=35+35)
    software-dependency-audit                  9% → 68%  (Δ=+59%, n=37+35)
    dapt-intrusion-detection                   26% → 84%  (Δ=+58%, n=37+35)
    simpo-code-reproduction                    0% → 57%  (Δ=+57%, n=35+35)
    lake-warming-attribution                   14% → 65%  (Δ=+51%, n=37+35)

  这 12 个 task 的共同特征：都需要 agent 具备特定领域的程序性知识——如何解析
  SEC 财务报表（sec-financial-report）、如何做地震板块计算（earthquake-plate-
  calculation）、如何用 Trivy 做依赖审计（software-dependency-audit）。没有
  skill 时，模型根本不知道该使用什么工具和方法，pass rate 接近 0%；有了 skill
  后，一条关键指令（如「使用 Trivy 扫描并输出 CSV」）就把模型拉到了正确轨道，
  pass rate 跃升至 60-86%。

  唯一的 negative task：
    taxonomy-tree-merge                        39% → 0%  (Δ=-39%)

五、Anchor 假设的五项证据
------------------------------------------------------------------------------

  证据 1：Δ 分布呈 bimodal

    如果 skill 的作用是弥散的（每步稍好），预期 Δ 呈正态分布，集中在均值附近。
    如果是 anchor 的（要么命中要么不命中），预期 Δ 呈 bimodal——大量 task
    接近零（skill 与 task 不匹配），少数 task 有大幅跳升（skill 命中 anchor）。

    实际观测（SkillsBench 84 tasks）：
      |Δ|≥50%（极端值）：     12 tasks
      |Δ|<5%（接近零）：      31 tasks
      5%≤|Δ|<50%（中间值）：  41 tasks
      极端+零 = 43，中间 = 41

    直方图：
      [-1.0, -0.5)   0
      [-0.5, -0.2)  █ 1
      [-0.2, -0.1)  █████████ 9
      [-0.1, +0.1)  ███████████████████████████████ 31
      [+0.1, +0.2)  ███████████████████ 19
      [+0.2, +0.5)  ████████████ 12
      [+0.5, +1.0)  ████████████ 12

    SWE-Skills-Bench 更极端：49 tasks 中 39 个 Δ≈0，7 个 Δ>0（最高+30%），
    3 个 Δ<0。完全没有中间值。

    结论：两个 benchmark 都支持 bimodal 分布，符合 anchor 预测。

  证据 2：Self-generated skill 完全无效

    SkillsBench self-generated skill 平均增益：-2.0%
    （对比 curated skill：+15.5%）

    这是区分 anchor 与 diffuse 的决定性证据。如果 skill 的作用是弥散的——
    即「提醒模型把事情做得更好」——那模型自己生成的 best practice 应该也有部分
    效果。但 self-generated skill 的增益为 -2.0%，甚至略有害。

    这说明 skill 的价值不在于「提醒」，而在于「告知」——告诉模型一个它不知道的
    关键信息。模型不知道的东西，它当然也生成不出来。这正是 anchor 的本质：
    信息要么被给予（anchor 命中），要么缺失（anchor 缺失），不存在弥散的
    中间路径。

  证据 3：Strong anchor task 的 no-skill pass rate 接近零

    12 个 strong anchor task 中：
      No-skill pass rate ≤ 15%：10/12
      No-skill pass rate ≤ 5%：  6/12

    模型在没有 skill 时几乎完全不会做这些 task，说明它缺少某个决定性的信息。
    Skill 提供了这个信息后，pass rate 跃升至 57-86%。这种「全无→全有」的
    翻转只可能来自 anchor，不可能来自弥散的渐进改善。

  证据 4：弱模型从 skill 中获益更多

    Haiku 4.5             baseline=11.0%  with-skill=27.7%  Δ=+16.7%
    Sonnet 4.5            baseline=17.3%  with-skill=31.8%  Δ=+14.5%
    Opus 4.5              baseline=22.0%  with-skill=45.3%  Δ=+23.2%
    Gemini 3 Pro          baseline=27.6%  with-skill=39.1%  Δ=+11.5%
    GPT-5.2               baseline=30.6%  with-skill=44.8%  Δ=+14.3%
    Opus 4.6              baseline=30.6%  with-skill=44.5%  Δ=+13.8%
    Gemini 3 Flash        baseline=31.3%  with-skill=46.5%  Δ=+15.2%

    Pearson r(baseline, Δ) = -0.342

    相关为负（r=-0.342），即 baseline 越低的模型从 skill 中获益越多。
    这与 anchor 假设一致：弱模型缺少更多 domain knowledge，因此有更多
    anchor 可以被 skill 命中。

    值得注意的是 Opus 4.5 是异常点：baseline 不高（22.0%）但 Δ 最大
    （+23.2%）。这可能因为 Opus 4.5 的指令遵循能力特别强——它更善于
    利用 skill 中的指令，即使自身 domain knowledge 不多。

  证据 5：Domain knowledge gap 与 skill 效果正相关

    SkillsBench 论文报告的按 domain 增益：
      Healthcare:          +51.9pp  （模型最不懂，anchor 最强）
      Energy:              +37.3pp
      Document Processing: +20.5pp
      Data Analysis:       +14.2pp
      Software Engineering: +4.5pp  （模型最懂，anchor 最弱）

    模型越不了解的领域，skill 的增益越大。在 SE 领域（模型训练数据最丰富的
    领域），skill 几乎无效——因为模型已经具备了大部分知识，没有 anchor 可以
    命中。在 Healthcare 领域（模型训练数据最少的领域），一条 skill 就可以
    让 pass rate 从接近零跃升 50 个百分点。

    SWE-Skills-Bench 从另一个角度印证了这一点：49 个纯 SE 任务中，只有 7 个
    skill 有效（14.3%），平均增益仅 +1.2%。在模型已经很强的领域，绝大多数
    skill 是冗余的。

六、Anchor 模式的完整图景
------------------------------------------------------------------------------

  综合以上证据，Skill 对 Agent 的作用机制可以描述为：

  1. Skill 的作用是局部的、集中的，而非全局的、弥散的。它在 Agent 决策树的
     某个高层节点上提供了关键信息，使 Agent 选择了正确的分支。后续步骤是
     Agent 在正确分支上的自然展开，不是 skill 的持续影响。

  2. Skill 的有效性是 task-dependent 的。对于 133 个 task 中的约 52.6%，
     skill 完全无效（|Δ|<5%）；对于 23.3%，skill 提供了决定性帮助。
     这说明 skill 与 task 之间存在匹配关系——只有当 skill 恰好包含了 Agent
     缺失的那个关键知识点时，它才发挥作用。

  3. 错误的 skill 可以是有害的（3.0% 的 task），因为它把 Agent 锚定在了
     错误的方向上。这是 anchor 模式独有的风险——弥散影响不会导致如此剧烈
     的性能下降。

  4. 模型自身的 domain knowledge 是关键调节变量。模型越不了解某个领域，
     skill 的 anchor 效果越强（Healthcare +51.9pp vs SE +4.5pp）。
     这意味着 skill 的价值 = 它弥补的 knowledge gap 的大小。

七、对 Skill 设计的实践启示
------------------------------------------------------------------------------

  1. 质量 >> 数量。大部分 skill 是无效的。与其生成 10,000 条 skill，不如
     精心策划 100 条，确保每条都包含模型不知道的关键信息。

  2. 针对模型的 knowledge gap 设计 skill。先评估模型在目标领域的 baseline
     能力，找到它「不知道什么」，然后只针对这些 gap 写 skill。给模型
     「它已经知道的东西」是浪费。

  3. Focused > Comprehensive。SkillsBench 发现「2-3 modules 的 focused
     skill」优于「面面俱到的 comprehensive documentation」。这也符合 anchor
     模式——Agent 需要的是一个精确的关键指令，不是一本百科全书。

  4. 警惕 skill 的负面效果。3% 的 task 中 skill 有害。在部署 skill 前应
     在 held-out tasks 上做验证，确保 skill 不会把模型带到错误方向。
